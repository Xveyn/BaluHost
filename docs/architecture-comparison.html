NotificationRepositoryImpl.kt Kotlin 
// Mobile: Cache-first + Network-Sync
override fun getNotifications(
  forceRefresh: Boolean
): Flow<List<Notification>> = flow {
  // 1. Sofort lokale Daten zeigen
  val cached = notificationDao.getAll()
  emit(cached.map { it.toDomain() })

  // 2. Im Hintergrund Server fragen
  if (forceRefresh || isCacheStale()) {
    try {
      val remote = api.getNotifications()
      notificationDao.insertAll(
        remote.map { it.toEntity() }
      )
      // 3. Aktualisierte Daten emittieren
      emit(remote.map { it.toDomain() })
    } catch (e: Exception) {
      // Offline? Kein Problem, Cache reicht.
    }
  }
}

4. Warum braucht Mobile so viele Schichten?

Jede Schicht löst ein Problem, das Web-Apps nicht haben

Process Death

Problem: Android kann die App jederzeit killen (RAM-Druck).
Web: Gibt es nicht — Tab offen = läuft.
Mobile: ViewModel überlebt Config-Changes, SavedStateHandle überlebt Process Death, Room DB überlebt alles.

Offline-Fähigkeit

Problem: Nutzer erwartet, dass die App ohne Netz funktioniert.
Web: Ohne Server → weiße Seite.
Mobile: Repository zeigt zuerst Room-Cache, synct im Hintergrund wenn Netz da.

Rotation / Config Change

Problem: Handy drehen = Activity wird zerstört und neu erstellt.
Web: Browser-Resize ändert nichts am State.
Mobile: ViewModel existiert außerhalb der Activity → State bleibt erhalten.

Push Notifications

Problem: App ist geschlossen, aber Nutzer soll benachrichtigt werden.
Web: WebSocket nur wenn Tab offen.
Mobile: FCM weckt die App auf → BaluFirebaseMessagingService → System-Notification.

Background Work

Problem: Upload läuft, aber Nutzer wechselt zur anderen App.
Web: Tab im Hintergrund → gedrosselt oder gestoppt.
Mobile: WorkManager garantiert Ausführung — auch nach Neustart.

Permissions

Problem: App braucht Kamera, Notifications, Dateizugriff.
Web: Browser fragt automatisch.
Mobile: Runtime Permissions mit explizitem Request-Flow, Fallback-UI wenn verweigert.

7. Lifecycle: Warum Mobile so komplex ist

Was passiert, wenn der Nutzer die App öffnet, dreht, wechselt, und zurückkehrt?

🌐 Web Lifecycle

1
Tab öffnen React mountet → useEffect feuert → API Call
2
Daten anzeigen setState → Re-Render → DOM Update
3
Tab wechseln Bleibt im Hintergrund, Timer werden gedrosselt
4
Tab schließen useEffect Cleanup → alles weg. Kein State, kein Cache.
Einfach. Mount → Run → Unmount. Das war's. Kein Process Death, keine Rotation, keine Permissions. Der Browser kümmert sich um den Rest.

📱 Android Lifecycle

1
App öffnen Activity.onCreate → Compose setzt auf → ViewModel initialisiert
2
Daten laden ViewModel → UseCase → Repository (Room zuerst, dann API)
3
Handy drehen Activity.onDestroy → Activity.onCreate (NEU!) → ViewModel lebt weiter
4
App wechseln onPause → onStop → ViewModel und Room bleiben
5
Android killt die App (RAM) Process Death! ViewModel weg, aber SavedStateHandle + Room DB überleben
6
App wieder öffnen Neuer Process → Room liefert gecachte Daten → API synct im Hintergrund
7
Push empfangen (App zu) FCM → FirebaseMessagingService → System Notification
Komplex. Die Activity kann jederzeit zerstört werden. State muss in verschiedenen "Überlebensstufen" gehalten werden: ViewModel (Rotation), SavedStateHandle (Process Death), Room DB (alles).

8. Zusammenfassung

Das Gesamtbild auf einen Blick

Web App = Fenster zum Server
React (Thin Client)
Zeigt an, was der Server liefert.
State: useState — flüchtig.
Cache: keiner. Offline: unmöglich.
↓
FastAPI (Fat Server)
Auth, Business-Logik, Datenbank —
alles passiert hier. Der Server ist die
Single Source of Truth.
Mobile App = eigenständige Anwendung
Kotlin (Fat Client)
Eigene Datenbank (Room), Offline-Queue,
Background Worker, Push Notifications.
Funktioniert auch ohne Server.
↕ Sync wenn online
FastAPI (gleicher Server)
Gleiche API-Endpoints, gleiche Auth.
Aber die App ist nicht abhängig —
sie synchronisiert sich.

Der Kernunterschied: Die Web-App ist ein Fenster — ohne Server ist sie nutzlos. Die Mobile-App ist eine eigenständige Anwendung mit lokaler Datenbank, die sich mit dem Server synchronisiert. Deshalb braucht sie ~4x so viele Dateien pro Feature, dafür funktioniert sie auch im Tunnel, im Flugzeug, oder wenn der Server down ist.
