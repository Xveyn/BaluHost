Executive Summary

BaluHost is an ambitious and feature-rich NAS management platform with strong architectural foundations including a well-organized monorepo, clean hardware abstraction via Protocol patterns, a mature plugin system, multi-worker coordination, and device-aware frontend builds. The project has 1,322 backend tests and comprehensive security measures (JWT, 2FA, RBAC, rate limiting, audit logging). However, the audit reveals critical gaps: 5 critical security vulnerabilities (password validation bypass, open registration, hardcoded admin password, command injection surface, plaintext VPN keys), no CI execution of any test suite despite having 1,500+ tests, a critical N+1 query pattern generating 300-400 queries per file listing request, and significant coding practice debt (532 silent exception handlers, DRY violations). The overall score of 64/100 (D) reflects a project with excellent architecture and ambition that needs focused attention on security hardening, CI/CD test automation, performance optimization of the file listing pipeline, and code quality improvements.

  2. CRITICAL: Restrict user registration to admin-authorized paths (invitation tokens or admin approval) to prevent unauthorized account creation
  3. CRITICAL: Add a production validator for the default admin password ('DevMode2024') and force password change on first login
  4. CRITICAL: Fix the N+1 query pattern in directory listing by pre-fetching metadata, ownership, and share data in bulk queries
  5. HIGH: Add GitHub Actions workflows to run pytest and Vitest on every push/PR — 1,500+ tests currently run only locally
  6. HIGH: Move JWT tokens from localStorage to HTTP-only, Secure, SameSite cookies to mitigate XSS token theft
  7. HIGH: Remove 'unsafe-inline' and 'unsafe-eval' from production CSP and make CSP conditional on environment
  8. HIGH: Extract all 174 direct database calls from route handlers into corresponding service classes per documented coding standards
  9. HIGH: Increase default Uvicorn workers from 1 to match CPU cores for production Docker deployment
  10. MEDIUM: Globally replace 90+ datetime.utcnow() calls with datetime.now(timezone.utc) before Python 3.12 deprecation
  11. MEDIUM: Create a session management context manager to eliminate 17+ duplicated should_close boilerplate patterns
  12. MEDIUM: Implement application-level response caching for frequently polled monitoring endpoints
  13. MEDIUM: Add API versioning (/api/v1/) before breaking changes affect the three client platforms (web, desktop, mobile)
  14. MEDIUM: Convert 12 'assert True' placeholder tests in test_security_headers.py to actual header assertions

  16. LOW: Synchronize version numbers across __init__.py (1.8.2), pyproject.toml (1.12.0), and docs/ARCHITECTURE.md (1.4.2)

Coding Practices

D 69/100

The codebase demonstrates strong architectural patterns with good dependency injection, type hint coverage, and Pydantic validation. However, significant DRY violations (session management, audit logger, monitoring patterns), 532 silent exception handlers, and untyped dict parameters in security-critical endpoints undermine code quality.

HIGH Repetitive session management boilerplate

The 'should_close = db is None' pattern is duplicated 17+ times across users.py and metadata_db.py, and likely more throughout services. Every function repeats the same try/finally session lifecycle.

📄 backend/app/services/users.py

💡 Recommendation: Extract into a context manager: @contextmanager ensure_session(db=None) that yields a session and handles cleanup.

HIGH 532 bare except Exception handlers across 113 files

Many except Exception blocks silently swallow errors without logging. Examples in main.py (lines 164-165, 225-226) and auth.py (line 68-69) use 'pass' or return defaults, masking real bugs.

📄 backend/app/main.py

💡 Recommendation: Audit each handler. Replace with specific exception types where possible. At minimum add logger.debug('...', exc_info=True) to prevent silent failures.

HIGH Untyped payload: dict in route handlers bypasses validation

Change-password, refresh-token, and several file operation endpoints accept raw dict instead of Pydantic models, bypassing FastAPI's automatic validation and documentation generation.

📄 backend/app/api/routes/auth.py:430

💡 Recommendation: Replace all payload: dict parameters with proper Pydantic request models to restore validation and OpenAPI documentation.

MEDIUM Deprecated datetime.utcnow() used in 90+ locations

The codebase mixes deprecated datetime.utcnow() (returns naive datetimes) with modern datetime.now(timezone.utc) in newer code. Python 3.12 formally deprecates utcnow().

📄 backend/app/services/power/manager.py

💡 Recommendation: Globally replace datetime.utcnow() with datetime.now(timezone.utc) across all 90+ locations.

MEDIUM 1290+ f-string logger calls defeat lazy evaluation

Logger calls use f-strings instead of %s formatting, meaning strings are formatted even when the log level is disabled. The codebase inconsistently mixes both styles.

📄 backend/app/main.py:601

💡 Recommendation: Use logger.warning('msg: %s', var) consistently for lazy evaluation. Many linting tools can auto-fix this.

MEDIUM Duplicated audit logger initialization in route handlers

The line audit_logger = get_audit_logger_db() is called at the top of nearly every route handler across 10+ route files (47 occurrences).

📄 backend/app/api/routes/auth.py

💡 Recommendation: Make audit_logger a FastAPI dependency via Depends() to eliminate boilerplate.

MEDIUM Duplicated monitoring history endpoint pattern

CPU, Memory, Network, DiskIO, and Process history endpoints all repeat the same 30-line data source selection and fallback pattern with only the model class differing.

📄 backend/app/api/routes/monitoring.py

💡 Recommendation: Extract into a generic _get_history_with_fallback() function parameterized by collector name and model.

MEDIUM Inconsistent response formats

Some endpoints return raw dicts ({'message': '...'}) without response_model declarations, while others use proper Pydantic models. This creates inconsistent API documentation.

📄 backend/app/api/routes/auth.py

💡 Recommendation: Create standard response models like MessageResponse(BaseModel) and declare response_model on all endpoints.

MEDIUM Mixed Python Optional syntax across codebase

109 files use Optional[...] from typing while 37 files use modern X | None syntax. Many files mix both styles.

💡 Recommendation: Adopt X | None uniformly since the project uses Python 3.11+ and has from __future__ import annotations in several files.

MEDIUM TypeScript any usage in 56 locations across 19 files

The central API module (lib/api.ts) uses any in 5 places including CacheEntry type and memoizedApiRequest generic. Pi-hole components have 17 occurrences.

📄 client/src/lib/api.ts

💡 Recommendation: Replace any with unknown or proper generic types, especially in the API layer where response types can be constrained.

LOW German comments scattered in source code

German comments found in system.py, api.ts, and other files (e.g., 'Gibt aggregierte Speicherinformationen', '60 Sekunden').

📄 backend/app/api/routes/system.py:86

💡 Recommendation: Standardize all comments and docstrings in English for international maintainability.

LOW Dual secret keys create configuration confusion

config.py has both SECRET_KEY and token_secret with identical defaults and validators. The comment says token_secret is 'Legacy' but both must be set.

📄 backend/app/core/config.py:32

💡 Recommendation: Deprecate and remove token_secret if no longer used, or add a model validator that syncs them.

Security

F 53/100

The project shows strong security awareness with JWT implementation, 2FA, rate limiting, audit logging, bcrypt hashing, and path traversal prevention. However, 5 critical vulnerabilities (validation bypass on password change, open registration, hardcoded admin password, command injection surface, plaintext VPN keys) and 7 high-severity issues (CSP weaknesses, localStorage tokens, encryption key reuse, CORS misconfiguration) require immediate attention.

CRITICAL Change-password endpoint lacks input validation

The /auth/change-password endpoint accepts a raw dict instead of a Pydantic model, completely bypassing password strength enforcement (min 8 chars, uppercase, lowercase, digit, common password blacklist). An authenticated user could set their password to 'a'.

📄 backend/app/api/routes/auth.py:427

💡 Recommendation: Create a ChangePasswordRequest Pydantic model that applies the same password validation as RegisterRequest.

CRITICAL Open user registration without authorization

The /auth/register endpoint is publicly accessible with no admin approval or invitation mechanism. Any external attacker who can reach the API can create user accounts with filesystem home directories and access to shared files.

📄 backend/app/api/routes/auth.py:369

💡 Recommendation: Require admin-issued invitation tokens for registration, or add a configuration flag to disable public registration entirely.

CRITICAL Hardcoded default admin password with no production validator

The admin_password defaults to 'DevMode2024' (intentionally chosen to bypass the password blacklist). Unlike SECRET_KEY, there is NO validator that rejects this default in production. The admin is silently created with this password on first startup.

📄 backend/app/core/config.py:47

💡 Recommendation: Add a Pydantic validator that rejects default admin passwords in production mode. Force password change on first admin login.

CRITICAL Command injection surface via username in Samba operations

The username parameter is passed directly to sudo useradd, smbpasswd, and other system commands. While asyncio.create_subprocess_exec prevents shell injection, the Samba functions accept any str without re-validating the username regex.

📄 backend/app/services/samba_service.py:52

💡 Recommendation: Add username validation directly in Samba service functions using strict regex. Never trust that callers have validated input.

CRITICAL VPN private key stored in plaintext in API response and database

WireGuard client creation returns the private key in cleartext in the response body and stores server_private_key in the database without encryption, while the Fritz!Box path properly encrypts.

📄 backend/app/services/vpn/service.py:180

💡 Recommendation: Encrypt all VPN private keys at rest using VPNEncryption.encrypt_key() before database storage. Mark config_content as one-time-display.

HIGH CSP allows unsafe-inline and unsafe-eval in production

The security headers middleware applies 'unsafe-inline' and 'unsafe-eval' in the script-src CSP directive for all environments, with a comment noting it is for Vite dev server. This significantly weakens XSS protection in production.

📄 backend/app/middleware/security_headers.py:29

💡 Recommendation: Conditionally set CSP based on settings.is_dev_mode. Remove unsafe-inline and unsafe-eval from production CSP.

HIGH WebSocket authentication token passed as query parameter

The notification WebSocket endpoint accepts the full JWT access token (15-min TTL, full permissions) as a query parameter, where it gets logged in server access logs, proxy logs, and browser history.

📄 backend/app/api/routes/notifications.py:319

💡 Recommendation: Create a short-lived, scoped WebSocket connection token similar to the existing SSE token pattern.

HIGH JWT tokens stored in localStorage vulnerable to XSS

JWT tokens are stored in localStorage which is accessible to any JavaScript on the page. Combined with the CSP allowing unsafe-inline/unsafe-eval, any XSS vulnerability directly exposes the authentication token.

📄 client/src/lib/api.ts:155

💡 Recommendation: Use HTTP-only, Secure, SameSite cookies for token storage. Set tokens via Set-Cookie response headers.

HIGH VPN encryption key reused for TOTP secret encryption

The VPN_ENCRYPTION_KEY Fernet key is reused for encrypting VPN private keys, SSH private keys, and TOTP secrets. Compromise of one encrypted value's context leads to compromise of all.

📄 backend/app/services/totp_service.py:89

💡 Recommendation: Use separate encryption keys for separate concerns. Add dedicated TOTP_ENCRYPTION_KEY and SSH_ENCRYPTION_KEY configuration.

HIGH Overly permissive CORS configuration

CORS is configured with allow_credentials=True, allow_methods=['*'], and allow_headers=['*']. While origins are restricted, the wildcard methods and headers are overly permissive.

📄 backend/app/main.py:893

💡 Recommendation: Restrict allow_methods to specific HTTP methods used (GET, POST, PUT, DELETE, PATCH, OPTIONS). Restrict allow_headers to required headers.

HIGH Hardcoded database credentials in Docker and scripts

docker-compose.yml uses fallback passwords (changeme_in_production), setup scripts contain hardcoded credentials (baluhost_password), and pgAdmin/Grafana use admin/admin defaults.

📄 docker-compose.yml:14

💡 Recommendation: Remove default password fallbacks from docker-compose.yml. Require explicit environment variables for production credentials.

HIGH No CSRF protection with allow_credentials=True

Since CORS allows credentials and some authentication paths may use cookies, the application could be vulnerable to CSRF attacks on state-changing endpoints.

📄 backend/app/main.py:897

💡 Recommendation: Add CSRF token validation for state-changing operations, or ensure Bearer-only authentication with no cookie fallback.

MEDIUM Debug mode enabled by default

The config defaults debug=True and environment='development'. Anyone running the backend without explicitly setting these variables will have debug mode active, potentially exposing stack traces.

📄 backend/app/core/config.py:8

💡 Recommendation: Default debug to False. Require explicit opt-in for debug mode.

MEDIUM Brute-force tracking dict grows without bound

The _failed_login_attempts dict in auth.py has no maximum size. An attacker could flood with unique IPs to consume memory.

📄 backend/app/api/routes/auth.py:28

💡 Recommendation: Use cachetools.TTLCache with a maximum size, or an external store like Redis.

MEDIUM pgAdmin master password disabled in Docker

pgAdmin runs with PGADMIN_CONFIG_MASTER_PASSWORD_REQUIRED=False, meaning anyone accessing port 5050 gets full database management access.

📄 docker-compose.yml:138

💡 Recommendation: Enable master password requirement. Ensure pgAdmin port is not exposed to external networks.

MEDIUM Rate limiter uses in-memory storage only

The rate limiter uses memory:// storage, meaning each worker process maintains independent counters. Multi-worker deployments have ineffective rate limiting.

📄 backend/app/core/rate_limiter.py:52

💡 Recommendation: Use Redis-based storage for production rate limiting when multiple workers are deployed.

LOW Login endpoint vulnerable to timing-based user enumeration

The authenticate_user function returns early when user is not found (skipping bcrypt), creating a timing difference that could enumerate valid usernames.

📄 backend/app/services/auth.py:23

💡 Recommendation: Perform a dummy bcrypt hash comparison when user is not found to normalize timing.

LOW Sensitive information in debug log messages

Token subjects, usernames, and roles are logged at DEBUG level in deps.py. Debug log collection could expose user data.

📄 backend/app/api/deps.py:74

💡 Recommendation: Ensure production log level is INFO or higher. Consider redacting user identifiers in debug logs.

Performance

F 58/100

The most critical performance issue is the N+1 query pattern in directory listing, generating 300-400 queries per request. Combined with recursive folder size calculations and synchronous I/O in async contexts, file browsing can block the event loop for seconds. Single-worker Docker deployment, excessive frontend polling (8-12 intervals), and no response caching compound the throughput limitations.

CRITICAL N+1 query pattern in directory listing generates 300-400 queries per request

list_directory() iterates over every filesystem entry and issues 3-4 individual database queries per entry (get_owner, is_path_shared_with_user, get_metadata, get_share_permissions). A directory with 100 files generates 300-400 SQL queries.

📄 backend/app/services/files/operations.py:118

💡 Recommendation: Pre-fetch all metadata, ownership, and share data for the directory in bulk queries before iterating. Build lookup maps from single queries.

HIGH Recursive folder size calculation blocks during directory listing

For every directory entry, get_folder_size() recursively walks the entire subdirectory tree using os.scandir(). First load of a directory with 20 subdirectories triggers 20 recursive filesystem walks, blocking the thread.

📄 backend/app/services/files/folder_size.py:19

💡 Recommendation: Compute folder sizes lazily (on demand), use stored sizes from database, or offload to background task and serve cached/estimated values.

HIGH Notification pagination fetches up to 10,000 rows to count total

To get the total count for pagination, the code re-runs the entire query with limit=10000 and uses len() on the result. This wastes database time and application memory.

📄 backend/app/api/routes/notifications.py:67

💡 Recommendation: Use a separate COUNT(*) query. Add a count_user_notifications() method to the service.

HIGH Synchronous file listing blocks event loop in async context

list_directory() is synchronous but performs heavy filesystem I/O and multiple database queries. When called from async route handlers, it blocks the event loop for seconds on large directories.

📄 backend/app/api/routes/files.py

💡 Recommendation: Wrap the synchronous call in asyncio.to_thread() or convert the route to a regular def function so FastAPI runs it in a threadpool.

HIGH Single Uvicorn worker in production Docker configuration

The Dockerfile CMD uses '--workers 1'. On a multi-core NAS system, this severely limits throughput. A single synchronous database call blocks all other requests.

📄 backend/Dockerfile:81

💡 Recommendation: Set --workers to min(2 * cpu_count + 1, 4) or make configurable via environment variable. The primary/secondary worker pattern already handles multi-worker correctly.

MEDIUM Notification count uses 9 separate database queries

get_unread_count endpoint runs a separate query for each of 8 notification categories plus 1 for total, producing 9 database roundtrips per request.

📄 backend/app/api/routes/notifications.py:105

💡 Recommendation: Use a single GROUP BY query: SELECT category, COUNT(*) FROM notifications WHERE ... GROUP BY category.

MEDIUM Multiple unbounded .all() queries without pagination

Several endpoints fetch all records without limits: sync state fetches ALL files for a user, VCL fetches ALL users, server profiles and Tapo devices have no pagination.

📄 backend/app/api/routes/sync.py:294

💡 Recommendation: Add pagination parameters (limit/offset or cursor-based) to list endpoints. For sync, implement delta sync instead of full state dumps.

MEDIUM 8-12 simultaneous polling intervals on Dashboard

When Dashboard is active, hooks set up 8-12 separate setInterval loops (monitoring, telemetry, services, plugins, activity feed, network status, energy), each making independent API requests.

📄 client/src/hooks/

💡 Recommendation: Consolidate polling into a single WebSocket connection for real-time data, or implement a single 'dashboard snapshot' API endpoint.

MEDIUM No application-level response caching

No caching layer exists for API responses. Every request hits the database or filesystem directly. Monitoring endpoints polled every few seconds would benefit from even 2-3 second TTL caches.

💡 Recommendation: Implement lightweight in-memory TTL cache (e.g., cachetools.TTLCache) for frequently polled endpoints like /monitoring/cpu/current and /system/telemetry.

MEDIUM Database stats function makes 3N queries for N metric types

get_database_stats runs count(), min(timestamp), max(timestamp) as 3 separate queries per metric type (5 types = 15 queries).

📄 backend/app/services/monitoring/retention_manager.py:184

💡 Recommendation: Combine into a single query per model using func.count(), func.min(), func.max() in one SELECT.

MEDIUM Memory buffer uses list.pop(0) which is O(n)

The monitoring collector's circular buffer uses list.pop(0) which requires shifting all elements. Called every 5 seconds for 4 collectors.

📄 backend/app/services/monitoring/base.py:128

💡 Recommendation: Replace List with collections.deque(maxlen=buffer_size) for O(1) append and automatic eviction.

MEDIUM Missing client_max_body_size in Docker nginx configuration

The nginx config does not set client_max_body_size (default 1MB), preventing file uploads larger than 1MB from reaching the backend which allows 10GB.

📄 client/nginx.conf

💡 Recommendation: Add 'client_max_body_size 10G;' to nginx.conf to match the backend's configured maximum.

MEDIUM Global in-memory caches without size bounds
